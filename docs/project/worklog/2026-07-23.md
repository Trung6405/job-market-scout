## 6. Gap-kind-classification spec

Traced a second class of false gap, distinct from the naming-variant one
fixed in `pipeline-hardening` phase 1: non-skill requirements (degrees,
experience thresholds, soft skills) are string-matched against the tech
stack, can never match, and so always surface as unmet gaps — e.g. a "STEM
degree in CS" requirement flagged missing for a profile holding a CS
degree. Brainstormed and spec'd the fix: tag each extracted requirement
with a closed-vocabulary `kind` and only gap-match the `skill` kind;
non-skill requirements render as display-only context.

| Path | Summary |
|---|---|
| `docs/agent/specs/gap-kind-classification/spec.md` | Problem, requirements, chosen approach (classify-by-kind), alternatives |
| `docs/agent/plans/gap-kind-classification/plan.md` | Three-phase plan, risks, blast radius, additive-migration rollout |
| `docs/agent/plans/gap-kind-classification/phase-1..3-*.md` | Task-level TDD breakdown: typed kind + matching → persistence → report rendering |

Spec approved and frozen; plan and all three phase docs written and
approved.

### Phase 1 — Typed kind + gap matching (complete)

| Commit | Summary |
|---|---|
| `feat(advisor): add RequirementKind vocabulary and RequirementItem` | Closed `RequirementKind` Literal (defaults to `skill`); `RequirementItem`; `kind` on `SkillGap` |
| `feat(advisor): type requirements as kinded items` | `must_have`/`nice_to_have` → `list[RequirementItem]` (`{name, kind}`) |
