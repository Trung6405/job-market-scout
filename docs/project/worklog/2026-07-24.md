## 1. Discord briefing verified live

First real run of the new Discord notification. The pipeline itself failed
earlier in the chain (see section 2), so the briefing was verified directly
with a throwaway smoke script that calls `send_message()` with a test embed.

Initial attempt returned `403 {"code": 50001, "message": "Missing Access"}`.
Probing the API isolated the cause: `GET /users/@me` returned 200 (token
valid, bot "Job Market Scout"), but `GET /channels/{id}` also returned 403 —
so the bot could not *see* the channel (50001), rather than merely lacking
permission to post (which would be 50013). The bot had not been added to the
server. After inviting it:

| Probe | Result |
|---|---|
| `GET /users/@me` | 200 — token valid |
| `GET /channels/{id}` | 200 — `daily-brief` |
| `POST /channels/{id}/messages` | 200 — embed rendered |

Note for future setup: a bot needs **Embed Links** (16384) in addition to
View Channel + Send Messages, or embed payloads are rejected.

No code changes — the notification path worked as written on the first live
attempt.

## 2. Advisor extraction truncation (`fix/advisor-extraction-batching`)

A full `docker compose up` run failed after scoring:

```
pydantic_core.ValidationError: 1 validation error for ListingRequirementsBatch
  Invalid JSON: EOF while parsing a value at line 882 column 37
```

**Root cause:** `run_requirements_extraction` built a single LLM call whose
response had to contain requirements for *every* listing (93 that day). No
`max_tokens` is set anywhere in the codebase, so the model's default output
cap applies; the JSON overran it and was cut mid-value, leaving invalid JSON
for `model_validate_json`. A deterministic output-length ceiling, not a model
quirk — it scales with listing count.

Worth noting the Scorer (`run_scorer`) has the identical single-call shape and
is vulnerable to the same failure at higher volumes; it survives today only
because a `ListingScore` is much smaller per listing than a
`ListingRequirements`. Left unchanged here to keep the fix to one concern.

| Commit | Summary |
|---|---|
| `fix(advisor): batch requirements extraction to avoid truncated JSON` | `REQUIREMENTS_BATCH_SIZE` setting (default 15); `run_requirements_extraction` splits listings into batches, one LLM call each, concatenating results; no call at all for an empty list |

Downstream needed no change: `agent.py` already filters to listings that have
extracted requirements and warns about any dropped, so partial results were
already tolerated.
